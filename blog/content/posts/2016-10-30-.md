This is the first project documented, an attempt to teach others how to use it. It is vastly different and more involved than writing function comments and in-code documentation. That is all low-level reference material and for the most part I keep up with that well enough.

I spent reading tips about writing good documentation. The **best** piece of advice I found, and what really stuck, is to approach documentation as a device to teach others. It just makes sense. Programmers read documentation to learn, therefore it follows that the purpose of documentation is to teach.

Great documentation has multiple layers. The top layer paints a picture, an overview, of the framework. Digging in deeper down, the bottom layer is reference material specific to each piece of treasure carried by the framework. The approach to documentation in TurtleBrains was to start at the details, write about every function, class, type and constant value that is part of the publicly facing API, what the purpose of each item is, constraints, and any other information to be aware of when using it.

This reference material was written directly in the TurtleBrains headers so that it would be easily accessible to write, and for others to read. A documentation generator can then strip the information and automate a portion of the documentation process. Doxygen has been praised by multiple projects, is free, and quite easy to use, so this is the chosen generator.

The next layer up from the reference material is still in the process of being written. It shows how objects relate to and interact with one another. This gives the programmer a general overview of what objects to create and how they interact with to meet the goals of the project.

Higher levels will get to code samples and snippets showing real-world usage examples, especially of common use cases. Provide these snippets for the potential programmers so they have something to look at before starting and more importantly give them a point to start from.
