Error handling is probably the most overlooked portion when programmers start writing a framework for others to use. When I set out to write TurtleBrains a primary objective with error handling was to remain truly flexible. An error, by the usage here, is the sort of failure that is an unexpected result. Such as loading an asset that is missing, or the data inputs are unexpected.

In a lot of game studios the approach on a missing asset is to crash as soon as possible when an error is encountered. Iâ€™ve grown to prefer this approach in my own games, as Iâ€™d rather have a crash letting me know that the texture or sprite failed to be created than to let the game continue with a white block representing the player, or worse nothing representing some object. However in the world of application development, exceptions are preferred as they can, if handled, continue on running like nothing happened. _I heard that moan from those with â€œexceptions are slow, we need performanceâ€ stuck in their minds._

TurtleBrains can handle these forms of errors in multiple ways, to allow the programmer or team using TurtleBrains to choose the way that works best for them. Do you want to halt the game, display an error screen with the line number, and failure message? Or would you prefer to catch an exception to be handled and continue on running. Want to do something different? Do it your way.

To achieve this, the core of TurtleBrains defines a pure virtual class, ErrorHandlerInterface, which can be overridden to perform as desired when an error condition is triggered. This interface will give you access to the file and line number, the error triggered on, as well as a formatted message string representing what went wrong. Throwing exceptions or using this custom interface can be toggled at runtime, so you will even be able to use exceptions in portions of code and not others, if desired.

The difference between the two, is tb\_error\_if will trigger when the test condition is true. This is a more natural and readable way to test and declare the failure points in code. Obviously a much better way to perform this function would be to make the numerator and denominator parameter types unsigned int instead of int **however this demonstrates the practice of defensive programming** without complications. If the programmer tried to call our PerformPositiveDivision() function with a _negative_ value for either numerator or denominator, an error condition would be triggered. TurtleBrains will then throw an exception or invoke the programmers customized error handler. The important part is to notify the programmer what went wrong so they can, with any luck, fix their inputs to be positive and successfully use the example function.
