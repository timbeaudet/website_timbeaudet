Recently work in [TurtleBrains][1] commenced on supporting game-pads, joysticks and steering wheels, aka; game controllers.Â During this process I&#8217;ve stumbled on an issue regarding the design of the public interface with the number of axes a controller supports/has. My steering wheel has 4 axes; wheel, accelerator, brake and clutch.Â The [DirectInput api][2] returns a structure separating the axes into different channels; X, Y, Z, Rx, Ry, Rz, Slider0 and Slider1. I guess this allows support for many controllers.

The choice that lay ahead is either ignore the number of axes a controller actually supports and letÂ users of the interface poll the separate channels, or remove the concept of channels and find a way to compress them down. This is actually possible with a bit of work behind the scenes with DirectInput as I can enumerate over each axis to see what channels are active though it wouldn&#8217;t be a 1 to 1. The question ultimately needs me to have more understanding with what is going onÂ behind the scenes for Mac and Linux platforms as well, it may just be that the TurtleBrains controller always reports all axes, even though some may never move.

It boils down to using the interface like:

[cpp]  
{  
//Some Code here.  
for (size_t axisIndex(0); axisIndex &lt; inputDevice.GetAxisCount(); ++axisIndex)  
{  
float axisValue = inputDevice.GetAxisValue(axisIndex);  
}  
}  
[/cpp]

Or like

[cpp]  
{  
float xValue = inputDevice.GetAxisValue(AxisX);  
float yValue = inputDevice.GetAxisValue(AxisY);  
float zValue = inputDevice.GetAxisValue(AxisZ);  
float rxValue = inputDevice.GetAxisValue(AxisRx);  
float ryValue = inputDevice.GetAxisValue(AxisRy);  
float rzValue = inputDevice.GetAxisValue(AxisRz);  
}  
[/cpp]

Either of them will work fairly well, and of course the second could be performed in a loop as the channels could be indexed 0&#8230;N, but N would notÂ necessarily represent the number of active axes the controller/device actually has. For instance my wheel would still return 8 axes channels even though only 4 are active. I_n my case it is **X**: steering, **Y**: accelerator, **Rz**: brake and **Slider0**: clutch._

It does involve more work, behind the scenes, to force X, Y, Rz, S0 to align with 0, 1, 2 and 3 axis indices.Â It seems to me a better use case since the channels, as I&#8217;ve been calling them, are a bit arbitrary anyway. Going withÂ axes indexed 0&#8230;N is also arbitrary but would at least have the convenience of dropping any unused axis.

Along with these public interface decisions comes with how it should be reported.Â I&#8217;ve always favored ease-of-use over most other conditions and that is certainly a desireÂ with the input device interface as well, however a floating number from 0 to 1 may not be precise enough, and so I may alsoÂ return a signed or unsigned short.Â I don&#8217;t think I&#8217;ve ever seen a controller that supports more than ~65k positions, so it should be fairly future proof _-these words will bite me someday-_Â and in the cases that only 256 values are returned it is easy to multiply into desired range without losing anything, as in the case of [XInput api][3] trigger values.
